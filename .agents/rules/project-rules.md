# 📜 قوانین پروژه Watashi Manager

این فایل حاوی تمام قوانینی است که در جریان توسعه با ایجنت قبلی وضع شده‌اند. من (ایجنت فعلی) موظفم همیشه به این قوانین پایبند باشم.

---

## قانون اول: به‌روزرسانی نسخه (Version Bump)
هنگام تغییر نسخه پنل (چه بتا و چه ریلیز)، حتماً باید **۴ فایل زیر هم‌زمان** آپدیت شوند و دقیقاً همان خط مربوطه تغییر کند:
1. `VERSION` (در روت اصلی یا مسیر مربوطه)
2. `hiddifypanel/VERSION` (در مسیر `hiddify-panel/src/hiddifypanel/VERSION`)
3. `hiddify-panel/src/hiddifypanel/VERSION.py` 
   - *نکته مهم:* در این فایل علاوه بر تغییر شماره نسخه، باید تاریخ و ساعت ریلیز (`__release_date__` و `__release_time__`) نیز به‌روزرسانی شود.
4. `pyproject.toml` (در مسیر `hiddify-panel/src/pyproject.toml`)

---

## قانون دوم: نسخه‌گذاری (بتا و ریلیز نهایی)
- پنل کلاً دو نسخه دارد: **ریلیز** و **بتا**.
- اگر آخر نام نسخه حرف `b` باشد (مثلاً `12.1.44b`)، نسخه **بتا** است.
- اگر آخر آن حرف `b` نباشد، نسخه **ریلیز نهایی** است. 
- *تبصره:* در صورتی که نسخه ریلیز نهایی باشد، متغیر `is_released_version` (در `VERSION.py` یا مشابه) باید به `True` تغییر کند.

## قانون چهارم: پروسه Push و Release
هیچ پوش (Push) یا ریلیزی بدون اجازه صریح کاربر نباید انجام شود. در صورت دریافت تاییدیه، دستورات همیشه از پترن ۳ مرحله‌ای زیر پیروی می‌کنند (کافیست نسخه و پیام کامیت تغییر کند):

```powershell
# مرحله ۱: پوش ساب‌ماژول پنل
cd d:\Downloads\Watashi-Manager\hiddify-panel\src
git add -A
git commit -m "نسخه و توضیحات"
git push origin dev

# مرحله ۲: پوش پروژه اصلی
cd d:\Downloads\Watashi-Manager
git pull origin main --rebase
git add -A
git commit -m "نسخه و توضیحات"
git push origin main

# مرحله ۳: ایجاد و پوش کردن Tag برای ریلیز
git tag -a v0.0.0b -m "v0.0.0b"
git push origin v0.0.0b
```

## قانون پنجم: ترجمه‌ها (Translations)
تغییرات مربوط به زبان‌ها باید با دقت کامل در فایل‌های زیر انجام شوند:

1. **فایل‌های JSON (۷ فایل در `translations.i18n/*.json`):**
   - دارای `description` و `label`.
   - کلیدها همیشه باید به ترتیب الفبایی باشند.

2. **فایل‌های PO (۶ فایل در `translations/*/LC_MESSAGES/messages.po`):**
   - زبان `fr` فایل po ندارد.
   - فرمت به صورت `msgid` و `msgstr` و پیشوند `config.` رعایت شود.
   - ترتیب کلیدها الفبایی و دارای کامنت ویژگی (`#`) باشد.
   - *نکته:* بعد از هرگونه تغییر، فایل‌های po حتماً باید به `.mo` کامپایل شوند.

3. **ارزش‌گذاری مقادیر:**
   - زبان `en`: حتماً باید متن کامل انگلیسی را داشته باشد.
   - زبان‌های `fa`, `zh`, `ru`, `pt`: در صورتی که ترجمه‌ای موجود نیست باید کلاً خالی بماند (`""`) تا بعداً توسط کامیونیتی پر شود.
   - زبان‌های `my` و `fr`: در صورت نبودِ ترجمه، باید مقدار انگلیسی به عنوان Fallback درونشان قرار بگیرد.
